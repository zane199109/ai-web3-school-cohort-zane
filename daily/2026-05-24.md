## 二、核心学习收获

### 2.1 受限 Web3 助手的核心边界

今天最重要的收获，是把“受限 Web3 助手”的职责边界进一步固化下来。

受限助手的核心不是“替用户完成链上操作”，而是“在高风险动作发生前提供足够清晰、可审计、可验证的辅助”。具体来说：
- AI 可以负责需求解析、风险提示、交易说明草稿、结果校验和复盘整理。
- 人工必须负责最终确认，尤其是钱包签名、授权、转账、合约写入等高风险动作。
- 所有涉及资产、权限和链上状态变化的步骤，都不能交给 AI 自动执行。

这让我更清楚地认识到：在 Web3 场景里，真正有价值的不是“全自动”，而是“可控、可审计、可验证”的自动化。

### 2.2 AI × Web3 相关概念的系统化整理

今天对一组基础但高频使用的概念做了重新梳理，并用自己的话重新表达。

其中最关键的几个认识是：
- LLM 不是事实库，而是语言生成与推理模型，必须结合工具和验证机制使用。
- Prompt 的核心不是“越长越好”，而是目标明确、约束清楚、上下文充分。
- Context Window 决定模型一次能看到多少上下文，超出窗口的信息可能会丢失，因此长任务需要外部文件或状态管理。
- Agent 不只是会聊天的模型，而是围绕目标进行规划、执行、检查和继续行动的系统。
- Guardrails 不能只停留在提示词层面，真正有效的边界必须结合权限控制、人工确认和日志审计。
- Human-in-the-loop 在高风险场景中非常重要，尤其是签名、授权、转账和发布操作，必须保留人工决策点。

这组概念的梳理，帮助我把“AI 能做什么”和“AI 不能做什么”区分得更清楚。

### 2.3 WCB 共学作业与笔记表达方式

今天还重新整理了 WCB 共学作业的表达方式，重点是把内容写得更像正式作业，而不是随意记录。

比较重要的格式原则包括：
- 标题清晰，层级分明。
- 内容围绕任务要求展开，不跑题。
- 用 Markdown 统一结构，便于复盘和提交。
- 尽量保留原始信息，但在表达上更专业、更规整。

这对后续写每日学习日志、任务总结、项目说明和提交材料都有帮助。

### 3.2 踩坑总结

1. 受限 Web3 助手的设计如果边界不清，很容易写成“看起来安全、实际上仍可能越权”的方案，因此必须把人工确认点写得足够明确。
2. 在整理作业文档时，如果只追求内容完整而忽略 Markdown 层级，会导致可读性下降，影响最终提交效果。
3. 测试网交易虽然金额很小，但仍然需要认真核对钱包地址、网络、交易哈希和浏览器结果，不能因为是测试网就放松确认。
