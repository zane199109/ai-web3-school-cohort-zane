## 二、核心学习收获

### 2.1 Prompt 不是 Policy，资金安全必须下沉到底层约束

今天最大的收获是：当 Agent 只负责回答问题时，Prompt 可以作为行为引导；但当 Agent 开始移动资金时，自然语言提示词不再具备强制约束力。

如果只依赖 Prompt，Agent 可能因为提示词注入、模型幻觉、恶意网页内容或不完整上下文，发起超出用户真实意图的交易。资金安全不能建立在“模型应该听话”的假设上，而要建立在密码学、策略引擎和合约可校验规则之上。

Cobo 的关键价值在于把“人类授权”从应用层描述转化为基础层硬约束：

1. MPC 让任何单一主体都无法独立挪用资金。
2. Pact 把意图、执行计划、风控策略和完成条件编码为可执行边界。
3. Recipe 限制 Agent 必须沿着经过验证的链上操作路径执行，减少临时拼交易带来的风险。

### 2.2 Cobo 解决“人机授权”，ERC-8183/8004 补齐“机机交易”

Cobo 主要回答的是：Agent 会不会乱花主人的钱？它守住的是钱包出口和授权边界。

但 Agent 商业化还存在另一层问题：Agent 花出去的钱，是否换回了真实、合格、可验证的服务？例如 Agent 购买 API 数据，如果服务商返回垃圾数据，仅靠钱包策略并不能完成履约判断。

因此，Cobo 与 ERC-8183、ERC-8004 形成互补关系：

1. Cobo Pact 限制 Agent 最多能花多少钱、能调用哪些对象、授权多久后自动失效。
2. ERC-8183 通过托管、验收、结算、退款和 Slashing，约束服务商必须提供合格结果。
3. ERC-8004 将交易履约结果沉淀为可复用声誉，让未来交易可以读取历史表现。

这构成了一个更完整的 Agent 信任栈：Cobo 管“钱不能乱出”，8183 管“钱不能白出”，8004 管“坏服务商不能长期混”。

### 2.3 Recipe 与标准协议可以成为机器商业说明书

Cobo 的 Recipe 本质上是给 Agent 使用的“操作知识胶囊”，包含合约地址、ABI、参数边界、执行路径和安全注意事项。它说明了一个重要工程原则：不是让 LLM 自由发挥越多越好，而是把高风险动作收敛到已验证范式中。

同样，x402 支付与 ERC-8183 托管流程也可以被封装为专用 Recipe。Agent 遇到 HTTP 402 时，不需要临时理解支付协议，而是按固定流程完成：识别付费请求、组装支付头、提交资金到托管合约、等待服务返回、触发验收与结算。

这种方式把“机器可支付、机器可验收、机器可追责”的商业流程标准化，使 Agent 更适合参与真实经济活动。
