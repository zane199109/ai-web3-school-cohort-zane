## 二、核心学习收获

### 2.1 多模型协作模式

GLM 5.1 + DeepSeek V4 的组合模式效果显著：
- **GLM 5.1**：擅长长文本理解、架构讨论、评审意见整合，负责 PRD 和设计的骨架
- **DeepSeek V4**：响应速度快、代码细节准确，负责具体技术方案的细化和实现策略

两者交替使用，GLM 定方向 → DeepSeek 扣细节 → 人工评审 → 反复迭代，产出的文档质量远高于单一模型。

### 2.2 Cobo Agentic Wallet（CAW）架构理解

设计阶段重点研究了 CAW 的产品体系，发现和传统认知的 Cobo Portal 完全不同：

| 对比项 | Cobo Portal（第一印象） | CAW Agentic Wallet（实际） |
|--------|----------------------|--------------------------|
| 认证方式 | App ID + Secret → HMAC-SHA256 | X-API-Key 简单 Header |
| API Base | api.cobo.com/v2 | api-core.agenticwallet.dev.cobo.com/api/v1 |
| 钱包类型 | 托管钱包 | MPC TSS 钱包 |
| 本地组件 | 无 | cobo-tss-node（本地签名守护进程） |

这个认知差直接影响了设计和实现——如果按 Portal 的 API 写，后面全得推翻重来。

### 2.3 多 Agent 多钱包架构

设计文档明确了三 Agent 架构，每个持有独立 CAW 钱包：

```
Buyer  Agent → 独立 CAW 钱包（发榜锁定资金）
Seller Agent → 独立 CAW 钱包（接收结算）
Eval   Agent → 独立 CAW 钱包（仲裁 Gas 开销）
```

实现时发现 CAW dev 环境的一个坑：同一 API Key 下创建多个钱包会不稳定，最佳实践是一个 API Key 对应一个钱包。

### 2.4 全栈自动化的可行性

这次验证了一个高效的开发流程：
1. **多模型脑暴** → 产出结构化文档
2. **文档转指令** → 喂给 Hermes Agent
3. **自动执行** → Go 后端 + Solidity 合约 + React 前端
4. **人工验收** → 验证清单逐项确认

一天内完成了原本设计为 6 天的开发计划（虽然后端连 CAW 等外部依赖需等待环境就绪）。

## 三、实操与踩坑复盘

### 3.1 DeepSeek API Key 问题

Hermes 自身的 DeepSeek Key（`fe_oa_` 开头）是内部代理 Key，不能直接用于外部 API 调用。标准 DeepSeek API Key 应以 `sk-` 开头。
→ 待解决：需要去 platform.deepseek.com 创建正式 API Key。

### 3.2 WSL 环境限制
agent提供的解决方案会带来其他的隐患,需要重新解决
- **sudo 权限**：没有 passwordless sudo，系统级安装需交互密码。改用 `~/.local/bin/`。
- **Go 网络**：`sum.golang.org` 不可达。使用 `GONOSUMCHECK=* GONOSUMDB=*` 跳过校验。
- **Go Linter**：子模块项目结构导致 linter 找不到 go.mod，不影响编译。

### 3.3 CAW Wallet 创建

TSS Node 初始化需要交互式密码（至少 16 位）。解决：使用 `--key-file` 参数绕过。
Dev 环境下钱包创建较慢（MPC 密钥生成），需要等待几分钟才能 active。

### 3.4 终端 Security Masking

Hermes 终端会 Mask 敏感信息为 `***`，调试时难以确认实际值。通过 Python subprocess 或直接读文件可绕过。
