## 二、核心学习收获

### 2.1 Prompt Router 的产品与技术价值

今天完成的 Prompt Router Demo，本质上不是简单问答页面，而是一个小型 AI 学习 Workflow：用户只需要输入自然语言问题，系统先判断问题类型，再选择更合适的 Prompt 模板，最后调用 Hermes / OpenAI-compatible 模型接口生成回答。

这个设计让我更清楚地理解：AI 应用的关键不只是“调用模型”，而是把模型调用放进可解释、可控制、可验证的流程里。比如同样是 AI×Web3 学习问题，概念解释、技术原理、学习路径、项目实践和概念对比，对 Prompt 结构的要求完全不同。通过路由层先做问题分类，可以让模型输出更稳定，也更适合初学者继续学习。

### 2.2 Hermes 越用越强的机制

今天进一步梳理了 Hermes 的能力成长路径。Hermes 的长期价值不在于每次都重新开始推理，而在于把可复用经验沉淀下来：

1. memory 用来记录长期稳定的个人偏好、学习身份、环境事实和工作原则。
2. skill 用来记录可复用流程，比如课程作业生成、Demo 构建、GitHub 提交流程、WCB 笔记标准等。
3. session_search 用来按需检索旧对话，避免把大量历史长期塞进上下文。
4. cron 用来处理每日提醒、会议提醒、周期性任务检查等低频重复工作。
5. 工具调用用来替代模型“凭空猜测”，例如读取文件、搜索内容、运行测试和检查 Git 状态。

这说明 Agent 的成长不是玄学，而是由“记忆、技能、检索、自动化、验证”共同构成的工程系统。

### 2.3 降低模型消耗的关键思路

今天也复盘了 Hermes 模型消耗问题。主要结论是：真正高成本的部分通常不是输出太长，而是输入上下文过重。长会话、无关历史、过多技能、全量文件内容和重复解释，都会持续增加 input tokens。

后续更合理的策略是：普通整理任务使用低成本模型，复杂架构、难 Bug 和高风险 Web3 判断才使用高推理模型；需要旧信息时优先 session_search；代码任务只读取相关文件片段；任务完成后及时 /new，避免让一个会话无限膨胀。

### 3.2 踩坑总结

1. 本地环境最初没有安装 Go，直接运行构建失败。后续通过临时下载 Go 工具链完成验证，说明环境依赖要先检查，再进入构建流程。
2. Prompt Router 的推荐逻辑一开始没有正确处理大小写，导致 `Agent` 没有命中 agent 关键词。通过测试暴露问题后，补充了大小写归一化。
3. Memory 容量有限，不能把所有规则都直接新增。后续需要压缩已有记忆、去掉重复内容，再写入真正长期稳定的偏好。
4. Demo 涉及模型接口时，必须区分 `.env.example` 和真实 `.env`，公开仓库只能提交示例配置，不能暴露任何密钥。
