## 二、核心学习收获

### 2.1 文档复用与模板化思维

Week 2 深挖包中已经沉淀了大量 AEP 项目的核心信息——问题定义、架构设计、风险评估、验证计划。Week 3 的任务虽然数量多（18 个），但本质上是 **"同一组信息的不同切面"**：

- **Direction Card** = 项目的一句话快照
- **Proposal Memo** = 项目的 1 页简介
- **Repo Skeleton** = 项目的工程快照
- **Risk Memo** = 风险维度的深度展开
- **Sprint Plan** = 时间维度的深度展开
- **Cobo 赛道对齐** = 赞助商维度的深度展开

关键是**不要凭空写 18 个独立文档**，而是维护一个"信息核心"，然后按任务需求裁剪出不同视图。这比独立写作节省了 70% 以上的时间。

### 2.2 WCB 提交标准理解

对比 Week 1 和 Week 2 的已通过提交，确认了 WCB 的审核偏好：

- **Proof 格式**：GitHub 链接到 markdown 文件即可，不需要截图（线上活动类除外）
- **审核粒度**：只要文档内容完整、结构清晰、覆盖任务要求的全部要点，一般都会通过
- **敏感信息红线**：绝对不能在 proof 中包含私钥、API Key、.env 文件内容——Week 1 曾有 REJECTED 记录
- **Repo Skeleton 的 6 要素**：Problem / Track / MVP Flow / Tech Stack / Risks / Validation Plan 缺一不可

### 2.3 多人多文档并行产出策略

18 个文档 + 1 个 README 在约 3 小时内完成，核心手法：

1. **按分值排序**：先写 40pts 的 Ready Pack（聚合文档），它本身就是其他文档的提纲
2. **交叉引用**：每个文档只写自己的独特内容，公共信息（项目名、赛道、目标用户）通过链接或简短说明引用
3. **批量写入**：使用程序化方式（execute_code + write_file）而非逐一手动写，减少重复劳动
4. **内容复用**：Risk Memo 复用 Direction Card 的风险表并展开；Cobo 对齐任务复用 CAW 集成计划的技术细节

## 三、实操与踩坑复盘

### 3.1 AEP-Hackathon Repo 非 Git 管理

**问题**：修改 AEP-Hackathon 的 README 后发现本地没有 `.git`，无法直接 push。
**原因**：AEP 项目可能在另一台机器上克隆，本地仅有工作文件。
**方案**：文件已修改好放在原位置，需要手动初始化 git 或拷贝到有 git 管理的目录再提交。

### 3.2 WCB 提交 API 探索

**问题**：尝试通过 REST API 直接提交 task proof，但 `tasks.submit`、`submission.create` 等常见 tRPC endpoint 均返回 NOT_FOUND。
**结论**：WCB 的提交功能只能通过 Web UI 完成，没有公开的 API 端点。自动化提交不可行，需要人工在 Dashboard 上操作。

### 3.3 文档结构设计取舍

**问题**：某些任务之间存在内容重叠（如 Proposal Memo 和 Direction Card 都有"目标用户"和"最小功能"），直接复制会显得冗余且有同步风险。
**方案**：Direction Card 做完整版，Proposal Memo 做精简引用版并指向 Direction Card，避免多文档维护问题。
