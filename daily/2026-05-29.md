## 二、核心学习收获

### 2.1 Agent Profile 的"九个要素"——从 Workflow 推导人格比从能力列表推导更可靠

**之前的做法**：先想"Agent 能做什么"，再写能力列表，最后画图。

**今天的纠正**：先画完整的 Workflow 时序图，再拆解关键节点，最后把每个节点映射到 Agent Profile 的九要素（Identity、Maintainer、Capability、Input/Output、Collaborators、Invocation、Pricing & Billing、Verification、Failure Handling）。这样做的好处是每个要素都有具体的业务流程做支撑，不会写出"假大空"的描述。

### 2.2 AI Agent 协作中"验证手册"的核心价值

**问题**：Agent 在写完代码后，视野局限在"编译通过、单元测试通过"的舒适区，不会主动连接 Anvil 跑端到端验证。

**具体案例**：Agent 写的 `GetJob` ABI 解析代码在 Go 单元测试里没问题，但连到 Anvil 上跑完整 Demo 就崩了，报 `improperly encoded uint8 value`。原因是合约启用了 `via-ir` 编译后，struct 返回值在 ABI 编码时前面多了一个 32 字节的动态偏移量（tuple pointer），普通的 `abi.Arguments.Unpack(result)` 不会自动处理这个偏移，导致所有字段位置错位。

**由此形成的协作规范**：
- 每个功能交付必须附带验证手册（用户自检清单）
- Agent 在交付前自己跑一遍验证手册的所有步骤
- 人工验证不仅是"用肉眼扫一眼代码"，而是执行一个预设的检查流程

### 2.3 三段式联调的"隐形收益"——Hash 对齐不是走过场

**Day1 手册写的三段式规则**：
1. 合约自测（Foundry 内部签名验签通过）
2. Hash 对齐（Go 与 Foundry 对同一消息的 Hash 逐字符一致）
3. 再上链联调

**实际验证**：Go 端 EIP-712 的 DomainSeparator、Permit structHash/digest、RejectAuth structHash/digest 五个 Hash 值与 Foundry 的测试输出逐字节一致。如果第三步出了问题，可以直接排除 Hash 编码问题，把排查范围缩小到"业务逻辑"或"交易构造"层面。

**教训**：测试数据必须使用跟合约测试完全相同的参数（同一个 buyer 地址、seller 地址、jobId、nonce、deadline），否则 Hash 对不上就是浪费时间排查。

### 2.4 EIP-712 签名编码中的"约定即接口"设计哲学

对比了一个有趣的设计选择：原来在 Solidity 里直接把 struct 暴露为公共函数签名 vs 用 EIP-712 的 typed data 做派生。Agent 选择后者——不是直接在 contract ABI 里写 `permit(buyer, seller, ...)`，而是先定义 `PermitMessage` 的结构体，用 `keccak256(abi.encode(PERMIT_TYPEHASH, ...))` 派生 structHash，再做 ECDSA 签名。

这样设计的价值在于：
- **CAW 金库不用信任 AEP 的业务逻辑**，只信任 AEP 的签名
- 只要正确实现了 domain 限制、deadline 和 nonce 重放保护，AEP 被攻破不会导致金库资金被盗
- 这是"安全边界分层"的思想——业务判决可以错，但签名验证不能绕过

### 3.2 踩坑总结

1. **合约 ABI 解析的 tuple 偏移问题**（我在 Day1 验证时发现）
   - **问题**：Go 后端 `GetJob` 因 Solidity struct 返回值的 ABI 编码带有 tuple offset 导致数据错位
   - **根因**：via-ir 编译或某些 Solidity 版本下，struct 返回值在 ABI 编码时前面有一个 32 字节的动态偏移指针，`abi.Arguments.Unpack` 不自动处理
   - **纠正方式**：先读取前 32 字节的 offset，`payload = result[offset:]`，再对 payload 做 Unpack
   - **教训**：所有合约读取方法的 Go 实现都要在 Anvil 上做端到端验证（connect to real chain），不能只靠单元测试

2. **Agent 跳过端到端验证**
   - **表现**：Agent 每次写完代码只跑 `go build` 和 `go test`，不主动连接 Anvil 跑完整 Demo
   - **我的要求**：每个功能写完必须连 Anvil 跑一遍完整 Demo 来验证
   - **解决方案**：要求每次功能交付附带验证手册，Agent 交付前自己先跑一遍

3. **验证手册的 Step 9/10 要用真实的链上数据验证**
   - **问题**：最初写的验证手册只有"部署→测试"的粗略步骤，没有具体的预期输出对照（比如 GetJob 返回什么 buyer/seller/state）
   - **纠正**：每个验证步骤都给出具体的预期输出格式，让用户可以逐行对照

4. **Go mod 版本管理**：`go tidy` 自动升级 `go-ethereum` 到 v1.17.3，而该版本要求 Go 1.24，跟环境的 Go 1.22 不兼容。需要手动固定版本为 v1.14.12。
