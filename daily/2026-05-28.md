## 二、核心学习收获

### 2.1 从"事后验货"到"全生命周期风控"——AEP 的认知升级

**之前的理解误区**：AEP 只是 ERC-8183 的 Evaluator（验收员身份），买方提交交付物，AEP 判断合格后放款。

**今天的核心洞察**：事后验货防不住"杀猪盘"。设想一个恶意服务商——先用 0.01U 的小单刷几百笔好评，累积高声誉，然后在一个 1000U 的大单中直接携款跑路。事后验货模式对此完全无力。

结论是：AEP 不能只做"验货员"，而要做贯穿交易全生命周期的**风控引擎**：

| 阶段 | AEP 的职责 | 技术实现 |
| :--- | :--- | :--- |
| **事前** | 风险评级 & 质押定价 | 读取 8004 声誉数据，计算质押率；签发 Permit 通过/拒绝 |
| **事中** | 实时监控 & 异常检测 | 监听链上事件，检测异常行为模式 |
| **事后** | 验收判定 & 罚没执行 | 规则引擎+LLM 双模验收；签发 Auth 触发结算或罚没 |

### 2.2 "防盗" vs "防骗"——AEP 与 Cobo CAW 的生态位边界

**困惑**：Cobo CAW（Cobo Agentic Wallet）已经有意图风控、MPC 共管、Pact 授权、Recipe 知识胶囊，AEP 是不是重复造轮子？

**边界分析**：

- **Cobo CAW** 解决的是"通用资金安全"——防黑客、防私钥泄露、防 Agent 被注入攻击后转走资金。它是**金库保安**，守住钱包出口和授权边界，确保"不该花的钱绝对花不出去"。
- **AEP** 解决的是"垂直业务质量"——防 AI 幻觉、防数据造假、防交了钱拿到空壳服务。它是**质检法官**，判断交付物的质量合格与否，确保"花出去的钱该换回合格的东西"。

**结论**：两者是共生关系，而非竞争关系。AEP 从 CAW 解决的资金安全出发，往上做了 CAW 不做的业务质量问题。CAW 解决"钱能不能安全出去"，AEP 解决"出去的钱是否换来合格服务"。

### 2.3 密码学预言机——打通 Cobo CAW 与 AEP 的关键设计

AEP 不依赖 CAW 开放定制 API，而是作为**密码学预言机**与 CAW 交互：

- **事前**向 CAW 签发 EIP-712 `Risk Permit`：包含风险等级、建议质押率、有效期限和 nonce。CAW 校验签名通过后，按要求锁定额外质押或直接放行。
- **事后**向 CAW 签发 EIP-712 `Reject Auth`：包含 Job ID、罚没金额、AEP 判决摘要。CAW 校验签名后执行资金划转——成功时释放给卖方，失败时罚没或退款。

**关键安全设计**：CAW 不信任 AEP 的业务逻辑（LLM 可能出错），只信任 AEP 的签名（ECDSA / EIP-712）。只要正确实现了签名验证、domain 限制、deadline 和 nonce 重放保护，AEP 被攻破不会导致 CAW 资金被盗，只会影响业务判决的正确性——这是"安全边界分层"的思想。

### 2.4 金额加权声誉——防操纵的算法设计

**痛点**：简单及格率（Pass/Task Count）极易被操纵。恶意服务商可以用 0.01U 的小单刷 100 单（全部通过），然后在一个 100U 的大单中诈骗。简单及格率 100% 完全看不出问题。

**算法**：

```
Weighted Pass Rate = 通过的累计金额 / 总累计金额
```

**效果**：

| 场景 | 简单及格率 | 金额加权评分 |
| :--- | :--- | :--- |
| 100 单 0.01U 全过 + 1 单 100U 失败 | 99% | 100×0.01 / (101×0.01+100) ≈ 0.99% |
| 10 单 10U 全过 | 100% | 100% |

在金额加权下，大单失败直接导致声誉断崖式下跌（从几乎 100% 暴跌到接近 0%），拉高下次交易的事前质押门槛。作恶成本远大于作恶收益。

## 三、实操与踩坑复盘

### 3.1 今日实操内容

1. **AEP 全链路架构推演**：从 x402 支付意图 → CAW 资金托管 → 8183 状态机 → AEP 验收签名 → 结算/罚没 → 8004 声誉上链，形成完整的"信任飞轮"闭环。
2. **协议比较分析深化**：x402 只管"怎么要钱"（HTTP 级的收银台），ERC-8183 只管"状态流转"（链上保险柜），ERC-8004 只管"链上档案"（声誉系统）。AEP 是它们的大脑和质检法官——让这些协议最终"长出了业务逻辑"。
3. **结合本周已有输出**：将 AEP 架构与之前 Week 2 的方向研究（week2-direction-research.md）、最小支付流程（week2-Minimal-Payment-and-Commerce-Flow-Breakdown.md）、威胁模型（week2-Agent-Workflow-Threat-Model-and-Confirmation-Strategy.md）进行了融合验证，确认 AEP 可以自然嵌入 Week 2 已分析的协议栈。

### 3.2 踩坑总结

1. **协议不是实现**：一开始容易陷入"AEP 定义了一堆协议标准"的误区。今天的推演让我重新明确优先级——没有跑通的实现，协议就是一张白纸。应该先做"重后端的参考实现"，让代码成为协议的 1 号证明，从 1 长出 N。
2. **声誉算法初期不需要太复杂**：金额加权是最小可行版本，不要一开始就引入时间衰减、多维度评分等复杂因子。
3. **安全设计不能等上线再补**：EIP-712 的 domain、deadline、nonce 这些保护措施，必须在第一版实现时就考虑，否则上线后补成本的代价极高。
