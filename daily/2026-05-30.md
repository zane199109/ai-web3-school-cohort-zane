## 二、核心学习收获

### 2.1 Agent Profile 九要素——从 Workflow 推导人格比从能力列表推导更可靠

Week 2 的 Agent Profile & Capability Claim 作业让我跳出「先想能力再画图」的惯性，改为「先画完整的 Workflow 时序图 → 拆解关键节点 → 把每个节点映射到九要素」的结构化方法。

**九要素框架的实际应用**：

| 要素 | 在 AEP 风控协同代理中的体现 |
|------|----------------------------|
| Identity | 代表买方利益的去中心化裁决与执行节点 |
| Maintainer | AEP 网络的专业节点运营商 |
| Capability | 数据获取与微支付、双重风控验收、密码学裁决 |
| Input/Output | 链上事件触发 → EIP-712 签名 + 链上 TxHash |
| Collaborators | Buyer/Seller Agent、Cobo CAW 金库、AEP8183 合约 |
| Invocation | 被动事件驱动 + 主动意图下发 |
| Pricing & Billing | 1% 托管费 + 10% 罚没分成 |
| Verification | 密码学不可篡改（ecrecover 校验 EIP-712 签名） |
| Failure Handling | 数据作恶→Slash、代理误判→仲裁罚没、链上拥堵→自动重试 |

**一个关键认知**：写九要素不是为了「填表格」，而是为了回答一个核心问题——这个 Agent 在什么条件下能做什么、不能做什么、失败了怎么办。每个要素都应该能追溯到 Workflow 中的具体节点。

### 2.2 A2A vs MPP 的生态位区分——协作通道 vs 资金通道

**A2A（Agent-to-Agent Protocol）** 解决的是协作与接口标准化问题——异构 Agent 之间如何发现、通信、对齐意图。在 AEP 场景中，买方通过 A2A 向 AEP 下发「请监控 Job #123」的意图，AEP 通过 A2A 返回「已接管并完成验收」。**A2A 只管对话，不管钱。**

**MPP（Micro-Payment Protocol / x402）** 解决的是机器级高频微支付问题——单次 API 调用只值 0.001 USDC，走传统 ERC20 Transfer 的 Gas 远超本金。MPP 通过在 HTTP 402 响应中嵌入金额和收款地址，让 Agent 可以在链下做秒级微支付，再批量结算上链。**MPP 只管钱怎么流，不管任务怎么协调。**

**协同关系**：A2A 是神经网络（传递意图），MPP 是毛细血管（微额供血），AEP + CAW 是免疫系统（风控裁决）。三者结合才构成可信的 Agent 商业闭环。

### 2.3 三层防护模型——ERC-4337 × Safe × Guard/Policy

Week 2 的 Permission Strategy 作业让我系统理解了 Agent 钱包安全的分层架构：

| 层级 | 机制 | 核心解决的问题 | 类比 |
|------|------|----------------|------|
| 账户层 | ERC-4337（账户抽象） | 密钥单点故障 & Gas 摩擦 & 自动化障碍 | 「能跑起来的基础」 |
| 所有权层 | Safe（多签智能账户） | 资金归属权 & 多签审批 & 权限分层 | 「钱归谁所有、谁有权批」 |
| 行为层 | Guard / Policy（Safe Guard + CAW Pact 等） | AI 幻觉 & 逻辑漏洞 & 越权操作 | 「跑得再快，也飞不出规则边界」 |

**重要启示**：以前的思路是「Agent 自己掌握私钥，自己做签名决策」，这是极度危险的。正确做法是 Agent 只负责「建议」，签名权由 CAW 的 MPC + Pact 策略引擎控制，资金永远在 Safe 里不在 Agent 手里。

### 2.4 x402 支付循环的实战验证——从概念到可运行 Demo

今天花了很多时间在 x402-caw-demo 上，核心目标是用最小成本验证 x402 + CAW 的真实支付循环能否跑通。

**架构**：
- **provider**（backend-go/provider/）：模拟一个带 HTTP 402 Paywall 的 AI Inference 服务，三个场景分别返回不同金额/收款地址的 402 响应
- **SimpleCAW 合约**（contract-foundry/src/SimpleCAW.sol）：链上策略执行器，本地预检通过后调用 `executePayment` 转账
- **CAW Policy Engine**（caw/policy.go）：链下策略检查，比较请求金额与 `MaxBudgetPerDay`、检查收款方是否在白名单内
- **Agent**（agent/main.go）：完整的 Agent 自动工作流——请求 API → 发现 402 → 本地 CAW 预检审计 → 链上支付 → 携带凭证重试

**三个自动轮询场景的验证结果**：

```
场景 1：合法请求（10 USDC → 白名单地址）
  → CAW Audit: APPROVE → 链上支付成功 → 获取数据 ✓

场景 2：钓鱼攻击（10 USDC → 未授权地址）
  → CAW Audit: REJECT (whitelist) → 本地拦截，不付 Gas ✓

场景 3：超额请求（200 USDC → 超出日预算 100 USDC）
  → CAW Audit: REJECT (budget) → 本地拦截，不付 Gas ✓
```

**踩坑记录**（下面 3.2 展开）：WSL 代理劫持、forge 版本输出格式变更、constructor-args 参数位置、USDC 精度（6 位）单位不统一。

### 3.2 踩坑总结

1. **WSL 代理劫持导致 forge/curl 连不上本地 Anvil**
   - **问题**：代理被 curl/forge 读取，导致请求 `127.0.0.1:8545` 时走了代理导致 502
   - **我的纠正**：脚本只 `unset http_proxy https_proxy`，但 curl 还读 `HTTP_PROXY HTTPS_PROXY ALL_PROXY no_proxy NO_PROXY`（大写版本）。必须把大小写全部清掉
   - **教训**：WSL 环境下所有涉及本地 JSON-RPC 的操作都必须确保代理被彻底清空。我要求脚本里加了完整的 unset 和 git config 的静默处理

2. **forge 1.7.2-nightly 输出格式变更**
   - **问题**：新版本 forge 输出 `Deployed to:` 而非之前的 `Contract created:`
   - **我的纠正**：Agent 写的脚本 grep 了旧格式，导致地址提取失败。我要求改为 `grep -oE 'Deployed to: 0x[0-9a-fA-F]{40}'` 再二次提取
   - **教训**：针对 CLI 工具的版本变更要做好防御性 grep，不能假设输出格式永远不变

3. **--constructor-args 参数位置敏感**
   - **问题**：新版本 forge 中 `--constructor-args` 如果放在 `--private-key` 和 `--rpc-url` 之前，clap 解析会把后面的 flag 也当作构造函数参数，报 `expected 1 but got 6`
   - **我的纠正**：必须把 `--constructor-args` 放在所有其他 flag 之后、命令结尾
   - **教训**：forge 的命令行参数解析对位置敏感，不是所有 flag 顺序无关

4. **USDC 精度单位不统一——所有 REJECT 的根因**
   - **问题**：真实 USDC 是 6 位精度（1 USDC = 10^6 wei），但 MockUSDC 合约默认继承了 ERC20 的 18 位精度。导致合约层记录的金额是 6 位格式（如 100 USDC = 100000000 wei），而 Go 代码里 Policy 和 Provider 还在用 18 位格式传值。两边对不上，比较永远为 true → 全部被 REJECT
   - **我的纠正**：显式覆写 MockUSDC 的 `decimals()` 返回 6；合约部署脚本中 `cast to-wei` 改为 `$((100 * 10**6))` 手动计算；Go 后端 `oneUSDC` 从 `10^18` 改为 `10^6`；Provider/Agent 的 Policy 精度同步调整
   - **教训**：Mock 合约默认精度可能不符合真实代币。用 OpenZeppelin 的 ERC20 时一定要显式覆写 `decimals()`，不能依赖默认值

5. **AI Agent 的脚本默认假设不适用于 WSL 环境**
   - **问题**：Agent 写的 `setup_and_deploy.sh` 假设标准 Linux 无代理环境，未考虑 WSL 的 Windows 代理穿透
   - **我的纠正**：多次要求修改脚本，逐步暴露了代理问题、forge 版本问题、constructor-args 问题
   - **教训**：WSL 代理环境是特殊的，所有涉及网络请求的脚本都应先清空代理环境变量

### 3.3 赛道公布后的方向反思

今天赛道公布的硬性变化是：**必须结合 Cobo Wallet 作为基础设施**。这意味着：

- 之前的 AEP 方案基于「自建 EIP-712 签名链 + MockCAW 日志体系」，CAW 只是概念引用
- 现在必须**真实接入 Cobo Wallet 的 API/合约**，包括 MPC 签名、Pact 策略、Recipe 知识胶囊
- 原有的 EIP-712 签名闭环设计依然有价值（密码学预言机思路），但对接锚点从「自建合约」变成了「Cobo 的合约接口」
- 需要重新理解 Cobo 的开发者文档和合约接口，评估哪些能力可以直接复用、哪些需要封装适配

这其实不是坏消息——Cobo CAW 的产品定位与之前我设想的「金库保安」角色高度一致，选择它作为基础设施反而验证了前几天的架构方向是对的。只是实现路径需要调整。
